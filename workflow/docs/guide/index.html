Get Started · 1

설치 후 첫 단계

한 줄 설치 스크립트로 템플릿이 내려왔다면, make setup → Claude 재시작 → OAuth 인증 → plm-hub:link 4단계로 본격 가동된다.

한 줄 설치 (이미 했다면 건너뛰기)

대시보드 Repo 탭 → "설치 스크립트 받기"에서 Keycloak 로그인 후 1회용 명령을 복사해 프로젝트 루트에서 실행. clone + 템플릿 + config + 토큰 슬롯이 한 번에 깔린다.

curl -fsSL '.../install/<token>' | bash

make setup — 플러그인 · MCP · 시드

플러그인 4종(plan·code·plm-hub·plm-channel)을 project scope로 설치, MCP 3종(chrome·context7·serena), .ouroboros 시드와 .env를 한 번에. 멱등하므로 재실행해도 안전.

make setup

(선택) make channels — 웹 [Sync] → 세션 push

웹 대시보드의 [Sync] 버튼이 이 터미널 세션으로 작업을 보내게 한다. managed-settings allowlist 기록에 sudo 1회 필요. make setup 중에 묻기도 한다.

make channels

Claude 재시작 → MCP 둘 다 인증

Claude Code를 다시 띄우고 프로젝트 신뢰를 승인. /mcp에서 ouroboros와 plm 둘 다 Authenticate(OAuth · Keycloak). 브라우저 로그인으로 자동 연결 — API Key 불필요.

/mcp → ouroboros → Authenticate · /mcp → plm → Authenticate

plm-hub:link — PLM 프로젝트 바인딩

로컬 워크플로우를 PLM 프로젝트에 묶는다. .ouroboros/config/plm.json 생성 + hook 푸시용 영속 토큰 자동 발급(승인한 OAuth를 재사용해 본인 역할 토큰 셀프 발급).

/plm-hub:link <project-id>
바인딩이 끝나면 기획 .md를 편집할 때마다 plm-sync hook이 PLM에 자동 upsert하고, 매 세션 종료 시 plm-gate hook이 게이트(G1~G3)를 경고로 표면화한다. 이후 평소 작업은 슬래시 커맨드만 쓰면 된다.
MCP URL은 인스턴스별로 다르다 — 설치 스크립트가 .mcp.json의 URL을 당신 인스턴스(전부 같은 k0s 클러스터)로 자동 치환한다. 아래 예시의 plm.shoi.ch 류는 기본 인스턴스 표기일 뿐이다.
Get Started · 2

두 레이어를 함께 깐다 — MCP + 플러그인

혼동하기 쉬운 핵심: MCP는 도구만 주고, 슬래시 커맨드·hook은 플러그인이 준다. 둘 다 있어야 워크플로우가 완성된다.

🔐

MCP 2종 — OAuth(Keycloak)

.mcp.json의 ouroboros(메모리/KG)와 plm(거버넌스). 사용자가 /mcp → Authenticate로 승인한 OAuth 세션으로만 동작. 셸·cron이 임의로 쓸 수 없다.

ouroboros — 메모리·지식·세션plm — 아티팩트·게이트·관계
🔑

hook — token (자동·무인)

무인 Edit/Stop hook은 .ouroboros/env/.env의 PLM_API_TOKEN으로 동작(plm-hub:link가 자동 발급). MCP를 호출할 수 없는 셸 레이어용. 토큰은 로그에서 마스킹.

plm-sync — .md 저장 시 upsertplm-gate — Stop 시 게이트 경고
권위 분담 — 본문·관계는 로컬 .md가 SSOT, Status·게이트·추적은 PLM이 권위. 역할이 겹치지 않으므로 충돌이 없다. 대시보드에서 사람이 바꾼 Status는 /plm-hub:pull로 로컬에 회수한다.

G1

요구 완전성

모든 SRS가 derives_from(→URS)를 보유하는가. 근거 없는 기능 요구를 표면화.

G2

설계 타당성

모든 SAD가 refs(→SRS)를 보유하는가. 요구에 닿지 않는 설계를 표면화.

게이트는 소프트(비차단) — Stop hook이 exit 0으로 경고만 띄운다. frontmatter엔 owner(작성) 관계만 적고, 역방향(피참조)은 매트릭스가 역산한다.

Traceability

코드 ↔ 요구 딥링크

구현 코드 위에 @plm <CODE> 주석을 달면, 요구에서 실제 코드 위치까지 끝단이 연결된다. 언어별 주석 문법 자유.

// @plm SRS-002  피드 생성 — fan-out-on-read
pub fn build_feed(user: UserId) -> Feed { /* ... */ }

그다음 /plm-hub:codescan을 돌리면 PLM에 Code 아티팩트(loc: path:line, build_state=as_built)와 realizes(Code→SRS/SAD) 관계가 생기고, 참조된 SRS/SAD의 .md frontmatter에 code_refs가 역기재된다.

SRS-002.md요구
←
realizesCode 아티팩트
←
@plm 주석loc: src/feed.rs:42
SRS에서 역추적 → Code → loc → 실제 코드 라인까지 도달
변경 파일 1건은 plm-codesync hook이 자동 동기한다. 전체 스캔 + GC(리네임/삭제 Code → Superseded)는 /plm-hub:codescan(수동). 죽은/벤더 코드는 .plmignore로 제외해 @plm 잔여 재생성을 막는다.

Reference

plm-hub 명령 레퍼런스

커맨드한 줄 설명
/plm-hub:link이 워크플로우를 PLM 프로젝트에 바인딩 + 영속 쓰기 토큰 자동 발급
/plm-hub:sync로컬 기획 .md 전체를 PLM과 일괄 동기(export/import)
/plm-hub:pullPLM 대시보드 편집분(Status·관계·code_refs·본문)을 로컬로 역방향 회수
/plm-hub:codescan@plm 주석 전체 스캔 → Code 아티팩트·realizes·loc·code_refs 동기 + GC
/plm-hub:gatesG1~G3 orphan과 재검토 큐(needs_review) 조회
/plm-hub:verify로컬↔PLM 동기 무결성 전수 재검증(드리프트 감사·자동 정합)
/plm-hub:agent빈약 본문 아티팩트를 work agent가 자동 보정(claude로 본문 생성→upsert)
/plm-hub:channel웹 [Sync] → 이 세션 push 채널 설정 안내(plm-channel)
/plm-hub:artifact-issue아티팩트(기획/코드) 발급 — MCP artifact_issue 호출
/plm-hub:artifact-get(project, code) 아티팩트 조회 — MCP artifact_get 호출
/plm-hub:relation-link두 아티팩트를 추적 관계로 연결 — MCP relation_link 호출
Optional

채널 — 웹 [Sync] → 터미널 세션

웹 대시보드의 [Sync] 버튼이 이 터미널의 Claude 세션으로 작업을 push하게 한다. Claude Code는 커스텀 채널 주입을 정책으로 막으므로, 마켓플레이스 플러그인 + managed-settings allowlist가 정식 경로다.

# 1) 플러그인 (make setup이 이미 설치)
claude plugin install plm-channel@jwk-platform

# 2) 채널 정책 활성화 (sudo·1회·멱등 — managed-settings allowlist)
make channels

# 3) 토큰 바인딩 (영속 토큰을 .env에 — 이미 link 했으면 됨)
/plm-hub:link <project>

# 4) 세션을 채널 대기 상태로 기동
claude --channels plugin:plm-channel@jwk-platform
세션을 띄워 두면, 웹에서 나를 assignee로 [Sync] enqueue 시 <channel source="plm" …>가 도착한다. 처리 후 Claude가 report로 회신. 세션이 닫히면 60s TTL로 offline.
Troubleshooting

막혔을 때

증상원인 · 해결
MCP가 needs authenticationOAuth 인증 전이라 정상. /mcp → plm(또는 ouroboros) → Authenticate 클릭. plm-hub:link에서 [skip] OAuth 토큰 없음이 나오면 인증 후 재실행.
플러그인이 stale / 옛 버전플러그인은 project scope가 우선이다. make refresh-plugins 후 세션에서 /reload-plugins. 옛 user-scope는 claude plugin uninstall <p>@jwk-platform --scope user로 정리.
슬래시 커맨드가 안 보임MCP만 깔고 플러그인을 안 깐 경우. make plugin-install 또는 make setup으로 plan·code·plm-hub 플러그인 설치.
URL이 안 맞음 / 연결 실패URL은 전부 당신 인스턴스의 k0s로 자동 치환된다. .mcp.json·.ouroboros/config/plm.json의 api_url이 당신 인스턴스를 가리키는지 확인.
대시보드 편집이 로컬에 없음plm-sync는 로컬→PLM 단방향. 대시보드 편집은 /plm-hub:pull로 회수해야 정합된다.
동기 누락이 의심됨/plm-hub:verify로 로컬↔PLM 전수 드리프트 감사·자동 정합. per-edit hook이 조용히 실패한 누적 desync를 잡는다.
채널 [Sync]가 세션에 안 옴make channels(sudo allowlist)와 --channels plugin:plm-channel@jwk-platform 기동이 모두 필요. 세션이 닫혀 있으면 offline.
Windows에서 설치 안 됨설치 스크립트는 Git Bash 또는 WSL에서. PowerShell에선 실행되지 않는다.
모든 hook은 exit 0 graceful — 동기·게이트가 실패해도 작업을 막지 않는다. 토큰이 없으면 동기화만 조용히 skip되고 로컬 기획·추적은 그대로 동작한다.
