# Knowledge Graph

Ouroboros의 Knowledge Graph(KG)는 Neo4j에 저장되며, 프로젝트의 문서 구조, 도메인 지식, 프로젝트 패턴을 그래프 형태로 관리합니다.

---

## 개요

Knowledge Graph는 메모리 시스템과 함께 Neo4j에 저장됩니다. 메모리가 **개별 경험**을 기록한다면, KG는 **일반화된 지식**을 구조화합니다.

| 구분 | 메모리 | Knowledge Graph |
|------|--------|-----------------|
| **성격** | 개별 경험 기록 | 일반화된 구조적 지식 |
| **예시** | "이 통계는 분기별 기준이다" | "경영진 보고서는 피라미드 구조를 사용한다" |
| **저장** | `memory_store` | `knowledge_store` |
| **변환** | `knowledge_distill`로 KG로 일반화 가능 | - |

---

## 노드 타입

KG에서 사용하는 주요 노드 타입입니다.

| 노드 타입 | 설명 | 예시 |
|----------|------|------|
| `Project` | 프로젝트 루트 노드 | marketing-report |
| `Document_Type` | 문서 유형 | 보고서, 백서, 기획서 |
| `Section_Pattern` | 문서 섹션 패턴 | 현황분석, 경쟁사비교, 전략제안 |
| `Writing_Pattern` | 작성 패턴 | 피라미드 구조, MECE 분류 |
| `Domain_Concept` | 도메인 개념 | 시장규모, ROI, 전환율 |
| `Convention` | 문서 작성 규칙 | 출처 표기, 용어 통일 |
| `Preference` | 사용자/프로젝트 선호 | "보고서는 결론부터 작성" |
| `Project_Knowledge` | 프로젝트 수준 지식 | 문서 포맷, 리뷰 기준 |
| `Technology` | 관련 기술/도구 | markitdown, PDF 변환 |

---

## 관계 (Relationship)

노드 간의 관계를 나타냅니다.

| 관계 | 설명 | 예시 |
|------|------|------|
| `CONTAINS` | 포함 관계 | Project → Document_Type |
| `USES` | 사용 관계 | Document_Type → Writing_Pattern |
| `RELATES_TO` | 관련 관계 | Domain_Concept → Domain_Concept |
| `PREFERS` | 선호 관계 | Preference → Convention |
| `SUPERSEDES` | 대체 관계 | Convention(new) → Convention(old) |
| `REFERENCES` | 참조 관계 | Section_Pattern → Domain_Concept |

### 관계 예시

```
Project:marketing-report
  ├─[CONTAINS]→ Document_Type:분기보고서
  │   ├─[USES]→ Writing_Pattern:피라미드구조
  │   ├─[CONTAINS]→ Section_Pattern:채널별ROI분석
  │   └─[REFERENCES]→ Domain_Concept:전환율
  ├─[CONTAINS]→ Document_Type:경쟁사분석
  │   └─[USES]→ Writing_Pattern:MECE분류
  └─[PREFERS]→ Convention:결론부터작성
```

## 네임스페이스

KG 데이터는 네임스페이스로 격리됩니다.

| 네임스페이스 | 범위 | 설명 |
|-------------|------|------|
| `project` | PROJECT_ID 단위 | 프로젝트별 문서 구조, 작성 패턴 |
| `personal` | DEVELOPER_USER_ID 단위 | 개인 선호, 작성 스타일 |
| `global` | 전체 | 범용 패턴, 도메인 지식 |

```
knowledge_store({
  type: "writing_pattern",
  name: "피라미드 구조",
  namespace: "project",
  properties: { ... }
})
```

## KG 도구

| 도구 | 설명 |
|------|------|
| `knowledge_store` | 노드 생성 |
| `knowledge_query` | 노드/관계 조회 (depth 지정 가능) |
| `knowledge_link` | 노드 간 관계 생성/수정 |
| `knowledge_distill` | 메모리 → KG 일반화 (DBSCAN 클러스터링) |

### knowledge_query 예시

```
knowledge_query({
  type: "writing_pattern",
  namespace: "project",
  depth: 2       // 2단계 관계까지 탐색
})
```

## knowledge_distill -- 메모리에서 KG로

`knowledge_distill` 도구는 축적된 시맨틱 메모리에서 일반화 가능한 지식을 추출하여 KG에 저장합니다.

```
knowledge_distill({ project_id: "my-project" })
```

서버에서 DBSCAN 클러스터링을 수행하고, 클라이언트에서 LLM 기반 일반화 텍스트를 생성합니다.

**변환 예시:**

- gotcha 3건 "통계 기준 시점 누락" → KG Convention "데이터 인용 시 기준 시점 명시 규칙"
- pattern 5건 "결론-근거-제안 순서 사용" → KG Writing_Pattern "경영진 보고서 피라미드 구조"

<Callout type="info">
  `knowledge_distill`은 유사한 메모리 그룹을 분석하여 공통 원리를 추출합니다. gotcha + pattern 합계 50건 이상일 때 효과적입니다.
</Callout>

## /onboarding과 KG

`/onboarding --mode project`를 실행하면 현재 프로젝트 구조가 KG에 반영됩니다.

- 문서 디렉토리 구조 → Document_Type 노드
- 기존 문서 패턴 → Writing_Pattern 노드
- 도메인 용어 → Domain_Concept 노드
- 사용자 선호 → Preference 노드
