# ADR-001 · 크로스플랫폼 프레임워크로 React Native 채택

## 맥락 (Context)
소규모 팀(부트스트랩)으로 iOS/Android를 동시에 도달해야 한다. 소셜·네트워크 효과는 도달 규모에 비례하므로(reference/hevy-success-analysis ③) 한 플랫폼만으론 불리하고, 벤치마크 Hevy도 React Native로 양 플랫폼을 동시 공략했다. 동시에 자동 카운팅(SAD-007) 같은 온디바이스 기능과 네이티브 결제·센서 접근도 필요하다.

## 결정 (Decision)
클라이언트를 **React Native 단일 코드베이스**로 구축한다. 네이티브 의존 기능(센서/카메라 추론·인앱결제)은 네이티브 모듈/브리지로 처리한다.

## 근거 / 대안
- 채택 이유: 단일 코드베이스로 양 플랫폼 동시 출시 → 소셜 도달·개발 효율 극대화. Hevy가 동일 스택으로 검증. 풍부한 생태계.
- 검토한 대안: ① **풀 네이티브(Swift+Kotlin)** — 성능·HealthKit 통합 우위지만 2배 개발비, 소팀 부적합. ② **Flutter** — 성능 좋으나 팀 RN/JS 친숙도·생태계(피트니스 라이브러리) 약세. ③ **iOS 우선 단일** — reference 권고이나 소셜 도달 손해 → 비목표 채택.

## 결과 (Consequences)
- 장점: 빠른 양 플랫폼 출시, 코드 재사용, JS/React 생태계 활용.
- 트레이드오프: 고성능 온디바이스 추론(카운팅)은 네이티브 모듈 추가 작업 필요, 일부 플랫폼 차이 핸들링.
- 되돌림 비용: 높음 — 후기 네이티브 전환은 사실상 재작성. 초기 결정이 중요(그래서 ADR).
