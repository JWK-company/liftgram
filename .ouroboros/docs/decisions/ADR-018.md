# ADR-018 · 백엔드 기술 스택·배포 — Node.js/TypeScript 모듈러 모놀리스 + PostgreSQL

## 맥락 (Context)
현재 앱은 완전 로컬(WatermelonDB·동기 0%)이라 소셜·DM·미디어·결제·크로스디바이스 동기 전부 **서버 백엔드 선행**이다. ADR-011은 토폴로지(모듈러 모놀리스)만 정했고 실제 구현 스택(런타임·프레임워크·DB·인증·호스팅)은 미결정 — 백엔드 착수의 전제다. 제약: 부트스트랩 소팀, 앱은 Expo/RN(TypeScript), 실시간 메시징·미디어/CDN은 이미 매니지드(ADR-015/016), 결제는 웹 MoR(ADR-013).

## 결정 (Decision)
백엔드를 **Node.js / TypeScript 모듈러 모놀리스(프레임워크 NestJS) + PostgreSQL** 로 구현한다.
- **인증**: 매니지드(Clerk·Supabase Auth 등) 또는 Keycloak(PLM에서 이미 사용)을 **어댑터로 추상화**해 교체 가능.
- **호스팅**: 매니지드 PaaS(Railway·Render·Fly.io)로 시작, 워커(트랜스코딩·정산 배치)는 분리.
- **위임**: 실시간 메시징·미디어/CDN·트랜스코딩은 ADR-015/016의 매니지드에 위임.
- **백엔드 코어 책임**: API·소셜그래프·피드/디스커버리·엔타이틀먼트·정산/배분(SAD-013)·알림 오케스트레이션. 모듈러 모놀리스 원칙(ADR-011)에 따라 도메인 모듈로 분리하되 단일 배포 단위 유지.

## 근거 / 대안
- 채택 이유: 앱(Expo/RN)과 **언어 통일(TypeScript)** → 풀스택 생산성·타입/검증 로직 공유·작은 팀의 낮은 컨텍스트 전환. PostgreSQL은 관계(소셜그래프·원장)+JSON 모두 강하고 성숙. 매니지드 인증/호스팅·실시간/미디어 위임으로 운영부담 최소.
- 검토한 대안: ① **Python(FastAPI/Django)** — 데이터/정산 생태계 강점이나 앱과 언어 분리(코드 공유·컨텍스트 손실). ② **Supabase 중심 BaaS** — 초기 속도 최대이나 User-centric 배분·글로벌 Payout 등 복잡 정산 로직에 별도 서버가 필요하고 벤더 종속 심화. ③ **Go** — 성능·동시성 우수하나 소팀 생산성·생태계·학습비용 부담·언어 분리. ④ **마이크로서비스** — ADR-011에서 이미 지양.

## 결과 (Consequences)
- 장점: 언어 통일(TS)로 빠른 개발·낮은 운영부담, 매니지드 위임, 모듈러 모놀리스 정합·점진 추출 가능.
- 트레이드오프: Node 단일 프로세스의 CPU 바운드 작업(트랜스코딩 등) 한계 → 매니지드(ADR-016)·워커 분리로 회피. 매니지드 인증/호스팅 벤더 종속 → 어댑터·표준(Postgres)으로 완화.
- 되돌림 비용: 중간 — 모듈러 모놀리스라 도메인 모듈을 서비스로 추출 가능, Postgres·TS 표준이라 핵심 자산 이전 용이.
