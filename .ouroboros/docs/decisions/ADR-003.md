# ADR-003 · 로컬 영속화 엔진으로 SQLite 기반 반응형 ORM(WatermelonDB) 채택

## 맥락 (Context)
로컬-우선(ADR-002)을 구현하려면 RN에서 수천 건의 세트 로그를 빠르게 읽고 UI에 반응적으로 바인딩하며, 동기 엔진과 잘 맞물리는 로컬 저장소가 필요하다(SAD-002·SAD-004). 세션 중 ≤3탭 성능과 대용량 이력 조회 성능이 동시에 요구된다.

## 결정 (Decision)
로컬 영속화 엔진으로 **SQLite 기반 + 반응형 동기 친화 ORM(WatermelonDB)**를 채택한다. 동기 엔진(ADR-002)은 이 저장소의 변경 추적 위에 구축한다. **웹 우선 타깃(ADR-012)**에서는 동일 WatermelonDB를 **LokiJS/IndexedDB 어댑터**(`app/src/db/adapter.web.ts`)로, 네이티브에서는 **SQLite/JSI 어댑터**(`app/src/db/adapter.ts`)로 구동한다 — 어댑터만 플랫폼 분기하고 스키마·모델·repository는 동일하다.

## 근거 / 대안
- 채택 이유: 대량 레코드에서도 lazy 쿼리로 빠름, RN 반응형 바인딩, 동기(pull/push) 프로토콜 내장에 가까움 → ADR-002와 정합.
- 검토한 대안: ① **순수 SQLite(직접)** — 제어 좋으나 동기·반응성 직접 구현 부담. ② **Realm** — 빠르나 동기는 유료 백엔드 결합 성향, 종속성. ③ **AsyncStorage/MMKV** — 키-값이라 관계형 이력 쿼리 부적합.

## 결과 (Consequences)
- 장점: 성능·반응성·동기 친화의 균형, 코어 로깅 UX 목표 충족.
- 트레이드오프: 라이브러리 추상화 학습곡선·스키마 마이그레이션 관리.
- 되돌림 비용: 중 — 데이터 접근 계층 캡슐화 시 교체 영향 국소화 가능.
