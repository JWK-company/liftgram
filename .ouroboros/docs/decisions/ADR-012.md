# ADR-012 · 1차 배포 타깃을 웹(PWA)으로 — Expo Web 단일 코드베이스, 네이티브 후순위

## 맥락 (Context)
제품 책임자의 실제 의도는 "웹으로 구현한 뒤 휴대폰에서도 웹기반 앱(PWA)으로 제공"이다. 그러나 ADR-001(React Native 채택)과 SAD-001("RN으로 iOS/Android 동시 제공")은 **네이티브 모바일 앱을 1차 배포 대상**으로 전제했다. 이 전제 차이로 Phase 0 구현을 실제 기기에 띄우려 할 때 네이티브 빌드 체인 마찰(full Xcode 부재, CocoaPods 미설치, WatermelonDB의 simdjson pod 중복 등)이 발생했고, 이는 "앱 스토어 배포 네이티브 앱"이라는 목표를 가정했기 때문이다. 한편 구현은 Expo로 되어 있어 동일 코드가 웹에서도 동작 가능하다는 점이 확인되었다.

## 결정 (Decision)
Phase 0의 **1차 배포 타깃을 웹(PWA)으로 확정**한다. **Expo Web(react-native-web)** 단일 코드베이스로 데스크톱/모바일 브라우저에 제공하고, 휴대폰에서는 PWA(홈 화면 설치)로 사용한다. 네이티브 iOS/Android는 **동일 코드베이스의 후순위(deferred) 옵션으로 유지**(폐기 아님)하며, 필요 시 `expo prebuild`로 빌드한다. 로컬 DB는 **WatermelonDB를 유지(ADR-003 불변)**하되 어댑터를 플랫폼별로 분기한다 — **웹 = LokiJS/IndexedDB**, **네이티브 = SQLite/JSI**.

## 근거 / 대안
- 채택 이유: ① 제품 의도(웹·PWA)와 정합. ② 앱 스토어 심사·네이티브 툴체인(Xcode/CocoaPods) 마찰 제거 → 배포 속도·반복 속도↑. ③ 기존 Expo/RN 구현(화면 12개·도메인·repository)을 100% 재사용 — 재작성 0. ④ 로컬-우선·오프라인(ADR-002)은 IndexedDB로 웹에서도 성립. ⑤ 네이티브 경로를 닫지 않아 향후 전환 옵션 보존.
- 검토한 대안: ① **네이티브 1차 유지(기존 ADR-001 전제)** — 제품 의도와 불일치, 스토어/툴체인 비용 큼 → 기각. ② **웹 전용 스택으로 재구축(Next.js 등)** — SEO·웹 UX 우위이나 화면 12개 재작성(대규모) + 네이티브 옵션 상실 → Phase 0에서는 과投資, 후속 검토 대상으로 보류. ③ **DB를 웹 네이티브 스토어(Dexie 등)로 교체** — ADR-003 변경·중복 작업 → LokiJS 어댑터로 최소 변경 달성하여 기각.

## 결과 (Consequences)
- 장점: Xcode/CocoaPods 불필요, 즉시 브라우저 실행·정적 호스팅(Vercel/Netlify/CF Pages)·PWA 배포 가능. 단일 코드로 웹+(후순위)네이티브. 빠른 반복.
- 트레이드오프: ① react-native-web에서 `Alert.alert`가 no-op → 확인/오류 다이얼로그를 웹용 모달로 교체하는 후속 과제(별도). ② SPA 특성상 SEO는 약함(SAD-001의 SEO 웹 표면은 별도 surface로 유지). ③ LokiJS는 SQLite 대비 대용량 성능 열위(Phase 0 규모에선 무영향). ④ 네이티브 결제(ADR-009 RevenueCat)·온디바이스 카운팅(ADR-005)은 네이티브 빌드 시에만 — 웹에서는 웹 결제/대체 경로 필요(후속).
- 되돌림 비용: 낮음 — 어댑터 분기만으로 양 타깃 공존. 네이티브 1차로 되돌리려면 빌드 체인 정비만 하면 됨(코드 변경 거의 없음).

## 연결
