# ADR-013 · 글로벌 결제·정산 모델 — 플랫폼이 Merchant of Record(웹 MoR 수금) + 유튜브식 구독풀 User-centric 트레이너 수익배분

## 맥락 (Context)
이 제품은 **양면 마켓플레이스**다 — 한쪽엔 운동하는 사용자(구매자), 다른 쪽엔 루틴·프로그램을 제공하는 트레이너(공급자). 따라서 결제는 단순한 "구독 수금"이 아니라, **(A) 사용자로부터 돈을 받고 (B) 트레이너에게 돈을 나눠주는** 양방향 정산(settlement) 문제다. 두 가지 제약이 모델을 강하게 한정한다: ① **전세계 사용자·트레이너**를 대상으로 하므로 수금 측 소비세(VAT/GST/Sales Tax, 구매자 국가 기준)와 지급 측 소득 원천징수(수취인 국가 + 미국 원천 기준)를 양쪽에서 처리해야 한다. 직접 100여 개국에 세무 등록하는 것은 소팀에 비현실적이다. ② 배포 면은 **웹/PWA 전용**(ADR-012)이라 Apple/Google In-App Purchase(15~30%) 강제를 회피할 수 있다. 한편 기존 ADR-009(구독 인프라 = RevenueCat)는 **iOS/Android 양 스토어 인앱결제**를 전제했는데, 이는 웹 우선(ADR-012) 전환과 정면으로 어긋나고, 트레이너 지급(payout) 축은 아예 다루지 않았다.

## 결정 (Decision)
글로벌 결제·정산을 **유튜브/스포티파이형 모델**로 확정한다.

1. **법적 포지션 — 플랫폼이 Merchant of Record(MoR, 판매자 본인).** 트레이너가 사용자에게 직접 판매하는 국내형 PG 분할정산이 아니라, 플랫폼이 매출을 소유하고 트레이너에게 **수익배분(revenue share)** 한다.
2. **수금(buyer side) — 웹 MoR 전문 사업자**(Paddle 또는 Lemon Squeezy/FastSpring)를 통해 전세계 소비세(VAT/GST/Sales Tax)를 자동 징수·납부한다. 수수료 ~5% 수준. 웹/PWA 결제이므로 앱스토어 IAP를 거치지 않는다.
3. **수익배분 구조 — 유튜브식 구독 풀(subscription pool).** 사용자 월 구독료가 플랫폼 매출이 되고, 풀에서 트레이너에게 분배한다.
4. **배분 알고리즘 — User-centric.** 구독자 1명의 (구독료 − 수수료)를 **그 사용자가 실제로 사용한 트레이너에게만** 인게이지먼트(해당 트레이너 루틴의 운동 세션/완주) 비중으로 배분한다.
5. **지급(payout side) — 글로벌 대량지급 사업자**(Trolley/Tipalti/PayPal Hyperwallet/Stripe Connect)를 이용한다. 트레이너 온보딩 시 세금양식(미국인 W-9, 외국인 W-8BEN)을 수집하고, 미국 원천 소득에 대한 원천징수(조세조약율 0/15/30%), 연말 세무서류(1099/1042-S), EU DAC7 보고를 사업자에게 위임한다.
6. **비용 원칙.** MoR·Payout 수수료와 플랫폼 take rate를 **수익배분율 안에 녹여**(유튜브: 45% 플랫폼 보유 / 55% 크리에이터 지급 참고) 트레이너 정산에서 공제 → 플랫폼 직접비용을 최소화한다.

## 근거 / 대안
- **채택 이유**: ① 글로벌에서 소비세·원천징수를 양쪽 모두 전문 사업자에게 위임 → 100여 개국 직접 세무등록 비용 회피(전문 사업자 수수료 ≪ 직접 컴플라이언스 비용). ② 웹 MoR(~5%)로 앱스토어 IAP(15~30%) 회피 — ADR-012(웹 우선)와 정합, 비용 최소화. ③ User-centric 배분은 "내가 따라 한 트레이너가 내 돈을 받는다"는 직관·투명성을 주고, 트레이너에게 *좋은 루틴 → 완주율 → 수익* 동기를 직접 연결(리텐션·품질 인센티브). ④ MoR가 판매자 본인이 됨으로써 트레이너 개개인이 글로벌 소비세를 처리할 필요가 없어진다.
- **검토한 대안**:
  - **(a) 국내 PG 분할정산(하위가맹·셀러정산)** — 국내 마켓플레이스 표준이고 자금 보유를 피해 규제상 깨끗하나, **전세계 VAT를 걷지 못하고 트레이너 개개인이 글로벌 소비세를 처리하는 것이 불가능**하여 글로벌에서 불성립 → 기각.
  - **(b) 네이티브 앱 IAP(Apple/Google)** — 스토어가 MoR로서 글로벌 소비세를 처리해 주지만 **15~30% 고비용**이고, 디지털 상품 판매 시 강제된다. ADR-012의 웹 우선과 배치 → 기각.
  - **(c) Pro-rata(스포티파이식) 풀 배분** — 전체 풀을 전체 인게이지먼트 점유율로 비례배분해 계산이 단순하나, **내 구독료가 내가 쓰지 않은 트레이너에게 흘러가** 인기 쏠림·투명성 저하·니치 트레이너 불리 → User-centric에 비해 공정성에서 열위로 기각.
  - **(d) 플랫폼 자체 정산대행(직접 자금 보유)** — 분할정산 추가 수수료는 아끼나 **에스크로 의무·전자금융업 라이선스·세무 인력**이라는 숨은 비용이 explicit 수수료를 훨씬 초과 → 기각.

## 결과 (Consequences)
- **장점**: 글로벌 세무(소비세·원천징수) 양쪽을 외부 위임하여 소팀이 컴플라이언스 부담 없이 전세계 판매·지급 가능. 웹 MoR로 앱스토어 30% 회피 → 비용 최소. User-centric 배분으로 공급자(트레이너) 품질·리텐션 인센티브 정렬. 플랫폼 직접 결제비용을 수익배분율에 흡수.
- **트레이드오프**: ① **ADR-009(RevenueCat) 대체** — 모바일 IAP 기반 구독 인프라 결정을 폐기하고 웹 MoR로 전환(향후 네이티브 출시 시 IAP 경로는 재검토 필요). ② **User-centric은 계산량·인게이지먼트 추적 부담**이 Pro-rata보다 크다(사용자별 사용량 원장 필요). ③ MoR 수수료(~5%)는 raw Stripe(~2.9%)보다 높으나 글로벌 세무 처리 비용을 포함한 값. ④ 수수료 전가율을 과도하게 높이면 **트레이너 이탈(공급 붕괴)** 위험 → take rate는 "트레이너가 남을 만큼"으로 균형 필요(비용 최소화 ≠ 마진 최대화). ⑤ 미국 원천징수·DAC7 등 규제 변화에 지속 대응 필요(사업자 위임으로 완화).
- **되돌림 비용**: 중 — MoR·Payout 사업자는 엔타이틀먼트/정산 추상화를 두면 교체 가능하나 마이그레이션 필요. 배분 알고리즘(User-centric↔Pro-rata)은 인게이지먼트 원장 설계에 종속되므로 사후 변경 비용이 큼 → SRS 단계에서 지표 정의를 신중히.

## 연결
- 영향 SRS/SAD: **SRS-014(수익화 — 수금 측: 웹 MoR 구독 수금·전세계 소비세)** 및 **SRS-016(트레이너 수익배분·정산·지급 — 지급 측: User-centric 배분·인게이지먼트 지표·take rate·정산 주기/지급 임계치·원천징수/세무서류·어뷰징 방지·정산 대시보드)**. 공급 측 이해관계자 추적은 URS-009·UCS-009로 신설.
- 대체(supersedes): **ADR-009**(구독 인프라 = RevenueCat) — 웹 우선(ADR-012) 전환으로 모바일 IAP 전제가 무효화되어 웹 MoR로 대체. 단 트레이너 지급(payout) 축은 ADR-009에 없던 신규 영역.
- 관련: ADR-012(웹 우선 PWA — 앱스토어 IAP 회피 근거), ADR-008(무료+그룹가치 수익화 — 상위 수익 전략과 정합).
