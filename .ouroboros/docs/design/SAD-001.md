# SAD-001 · 시스템 아키텍처 개요 & 기술 스택 (RN 크로스플랫폼)

## 개요 (Summary)
헬스앱은 **React Native 단일 코드베이스**로 iOS/Android를 동시 제공하는 **로컬-우선(offline-first) 클라이언트 + 클라우드 백엔드** 토폴로지를 따른다. 클라이언트는 네트워크 없이도 운동을 기록하고, 백엔드는 동기화·소셜 그래프·AI 오케스트레이션·결제 검증·콘텐츠(SEO)를 담당한다. 본 문서는 상위 토폴로지와 기술 스택 선택 근거만 다루며, 모듈 상세(SDS)와 각 도메인 설계는 SAD-002~010이 맡는다.

> **1차 배포 타깃 정정 (ADR-012)**: Phase 0의 1차 배포는 **웹(PWA) — Expo Web(react-native-web)** 단일 코드베이스로 한다. iOS/Android 네이티브는 동일 코드베이스의 후순위(deferred) 옵션이다(폐기 아님). 로컬 저장은 플랫폼 분기 — **웹=LokiJS/IndexedDB, 네이티브=SQLite/JSI**(ADR-003 유지, 어댑터만 분기). SEO 웹 표면은 별도 surface로 유지.

## 컴포넌트 / 책임
- **RN 클라이언트(App)**: UI·세션 상태·로컬 DB·온디바이스 기능(카운팅) 호스팅. 단일 코드베이스로 양 플랫폼 도달 극대화(Hevy 교훈 ③).
- **API 게이트웨이 / 백엔드 서비스**: 인증, 동기화 수신/병합, 소셜·그룹, AI 프로그래밍 오케스트레이션, 결제 엔타이틀먼트 검증을 도메인 서비스로 분리.
- **인증(Auth)**: 이메일/소셜 로그인, 세션 토큰 발급·갱신. 사용자 신원이 동기화·소셜·구독의 기준 키.
- **데이터 동기 계층(요약, 상세 SAD-004)**: 로컬-우선 저장과 서버 간 양방향 동기의 진입점.
- **콘텐츠/SEO 웹**: 운동 가이드·검색 유입을 위한 별도 웹 표면(저비용 획득 엔진).
- **관측/분석(Observability)**: 리텐션·전환·기능 채택 지표 수집(가드레일 내 비식별).

## 주요 결정 포인트 (→ ADR 후보)
- 크로스플랫폼 프레임워크로 React Native 채택(vs 네이티브/Flutter) → ADR.
- 백엔드 토폴로지(모듈러 모놀리스 vs 마이크로서비스) → ADR.
