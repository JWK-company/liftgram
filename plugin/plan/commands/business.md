# /business — 시장·경쟁 디스커버리

시장·경쟁을 **근거 기반**으로 파헤쳐 **차별화·수익화·진입여부(Go/No-Go)** 까지 판정한다(business/시장·경쟁 조사). `/plan` **선행(선택)** 스킬 — 기획을 추측이 아닌 근거에서 출발시킨다. 산출물 = **BS(선택) + 통합 Business**(구 MR+CA 병합 — ADR-027 · 비추적 · **프로젝트당 다수**(BIZ-00N — 조사는 반복된다)) **JSON 아티팩트(`CODE.json`)**:

| 코드 | 이름 | 파일 | 담당 |
|------|------|------|------|
| **BS** | Brainstorming(브레인스토밍) | `.ouroboros/docs/product/BS.json` | 아이디어 발산·초기 가설 (**선택** — 시드가 있으면) |
| **BIZ-00N** | 통합 조사 보고서(구 MR+CA) | `.ouroboros/docs/product/BIZ-00N.json` | 전문 보고서 흐름: 핵심요약→§1 시장(사이징)→§2 경쟁(경쟁표+체험루브릭)→§3 차별화→§4 수익모델→§5 손익(BEP)→§6 리스크→§7 로드맵→§8 Go/No-Go→부록·출처. **완성 보고서 HTML/PDF 파일 첨부가 본체**(plm_upload→file 노드) |

포맷 = `${CLAUDE_PLUGIN_ROOT}/templates/_ARTIFACT-JSON-FORMAT.md`(동형 — 웹 에디터·파일·DB가 같은 JSON). 섹션 scaffold·눈금 앵커 = `${CLAUDE_PLUGIN_ROOT}/templates/business.json`(구 mr/ca 병합 — 전문 보고서 흐름 §1~§8+부록·P1★/P2★ 티어). 완성 보고서 HTML 가이드 = `${CLAUDE_PLUGIN_ROOT}/templates/reports/business.md`. **markdown(.md) 아티팩트 생성 금지.**

> **디스패치**: `--feedback` **有 → Pass 2**(개발자 체험 피드백 반영·확정·판정 — 대상 Business 지정, 기본 최신) / **無 → Pass 1**(신규 BIZ-00N 발급 — 기존 조사가 있어도 새 회차는 새 Business).

## 제1원칙 (판정 규율)
1. **프레임워크가 판정을 강제** — 결정트리·눈금 앵커·근거셀. 감(感) 판정 금지.
2. **미출시 제품에 허위정밀 금지** — 각 게이트 입력에 **확신도{높/중/낮/근거불충분}**. 근거 없는 칸 = `N/A—사전단계`.
3. **게이트는 출시전 관측가능 입력만** — LTV/CAC 등 미관측치는 게이트가 아니라 **사후 검증 KPI**.
4. **증거 부재 = fail-safe** — 핵심 게이트입력이 근거불충분/저확신이면 Go도 No-Go도 확정 금지 → **판정유보(Pivot)**.

## 절차 — Pass 1 (조사 → Business 발급 → 체험 안내)
0. **캘리브레이션**: 카테고리(타깃 + JTBD + 폼팩터)로 직접 경쟁 3-8개 범위 고정 · 시장 스코프 · **생존선**(미정 → AskUserQuestion: 대략 입력 vs 기본가정[인디=라멘 수익성 / 펀디드=목표]).
1. **WebSearch**(출처유형 명시 · **2 독립출처 삼각검증** · 동일 원출처 = 1): 발견 · 대체재 · 수요/불만 · 공급측 가격 · **WTP(수요측)** · 수익화 · 플랫폼 수수료/규제/ATT. WTP 1순위 = 경쟁 실지불가(현시선호), reddit 감성 = 방향성·확신도 상한 '중간'.
2. **(선택) BS 발급**(시드 아이디어가 있으면 scaffold `${CLAUDE_PLUGIN_ROOT}/templates/bs.json` → `docs/product/BS-001.json` · `id="BS-001"`(P0-3 다수화 — 기존 BS 있으면 BS-002…) · `type="BS"` · `kind="idea"` 명시 · relations 없음): **권장 섹션**을 채운다 — **프로그램명(가칭) · 목적 · 사용자 종류(role) · 사용 흐름 · 수익화 포인트 · 로드맵 시드**. 특히 **"사용자 종류(role)"는 역할별 한 줄("역할명 — 목표")로** 적어 이후 role별 시뮬레이션/설계의 입력이 되게 하고, **"로드맵 시드"**(heading) 는 /plan의 RM 생성 근거(ADR-029 D2 전제)다. 시드가 없으면 생략.
3. **Business 발급**(scaffold `business.json` → `docs/product/BIZ-00N.json` · `id="BIZ-00N"`(기존 Business 다음 번호) · `type="Business"` · **relations: 입력 BS가 있으면 `{"generated_from":["BS-00N"]}`** — 생성 계보·P1-2): **Pass1 필수(P1★)** 섹션 골격을 채운다.
   - **§1 시장 개요(P1★)**: 사이징(TAM/SAM/SOM·필요점유율 밴드) · 수요 신호·트렌드 — 표+수치+출처, 소절 끝 시사점.
   - **§2 경쟁 분석(P1★)**: 경쟁표(§2.1 국내·§2.2 글로벌 + 우리행 · 직접≥3·간접≥1·대체재≥1) · §2.3 기능×경쟁 지형 매핑 · §2.4 체험 대상+루브릭 — 소절 끝 시사점.
   - **확증편향 방지**: 차별화(§3)/수익화(§4)는 이 단계에서 확정하지 말고 **가설 후보**로만. 근거 없는 칸 = `N/A—사전단계`. → 유효 JSON 자체 점검.
   - **완성 보고서 파일**: 조사 정리본 HTML(**`templates/reports/business.md` 가이드** — 예시 보고서 수준: 표지·목차·핵심요약·시장사이징·경쟁표·수익모델·BEP 손익·리스크·단계별 로드맵·Go/No-Go·참고 출처, **모든 수치에 출처·각 섹션 끝 시사점·3사이클 검증**)을 만들어 `/plm-hub:upload`(project 등록)→file 노드를 Business 본문 상단에 임베드 — 파일이 본체, 아티팩트 본문은 요약·판정.
4. **체험 안내 + 상태 전이**: 사용자에게 안내 — 추천 앱을 직접 써보고 **루브릭 채점(1/3/5) + "이 앱이 놓친 것 = 미충족 JTBD" 메모** → **대시보드 Business 본문 §2.5 "개발자 체험 피드백" 섹션 편집**(collab lease 409 회피 위해 **에디터 닫고**) → `/business --feedback`. `state.tasks.dig = {name:"디스커버리: {제품}", status:"awaiting-feedback", current_phase:"pass1"}` + `signal_record` + current.md 갱신.

## 절차 — Pass 2 (--feedback: 피드백 반영 → 확정 → 종합 판정)
1. **가드**: 대상 Business 없음 → 중단(먼저 Pass 1). 피드백 무변경 → AskUserQuestion(대기 vs 초안 진행).
2. **회수**: `/plm-hub:pull`(본문·개발자 편집분) + (comments_get 노출 시) 댓글.
3. **차별화 확정 (§3 · P2★)**: 각 후보 `{축 · 미충족JTBD · 방어가능성 · 타깃가치(체험근거)}` + 근거셀(§3.4). 좋은 차별점 = **(타깃 중시 ∧ 복제 어려움)**. 역량 부합 판단은 `/plan`으로 이관. **포지셔닝맵 = 좌표 테이블**(§3.4 · 대상 · 축1(1-5) · 축2(1-5) · 사분면 · 우리행).
4. **수익화 확정 (§4·§5 · P2★)**: **수익원 × 패키징 전술 2축**. 각 수익원 채택/기각 + **적합성 관측·추론 혼합축**(타깃WTP[추론] · 시장가격대 진입 · 전환마찰 · **단위기여마진**) 1-3 눈금 + 근거셀(§4.2). 억지 판정 금지("표준셋 + 카테고리 특수원", 폐쇄목록 아님). **가격 전략**(§4.3 · 경쟁가 분포 → 포지션·티어·무료 한계선·가격가설) · **차별점 ↔ 과금 매핑**(§4.4). **§5 비용·손익**: 인프라 비용·CAC/LTV·**BEP 시나리오 표**(유료 ARPU → 손익분기 사용자 → 필요 MAU)+해석까지 채워 수익화 가능성 판단.
5. **Go/No-Go/Pivot 종합 판정**(아래 §판정 게이트): **시장측(§1·§4·§5: 시장·수요·마진)+경쟁측(§2·§3: 방어가능 차별점) 양쪽 입력**을 결정트리에 대입 → 판정을 **§8 결론 · Go/No-Go/Pivot 섹션에 기록**(출시전 관측입력만 · N/A = fail-safe · 저확신 Go 금지).
6. **출력 계약 · 종단**: 결론 요약(상위3 차별점[CA] · 수익모델 스택[MR] · 수익 KPI 벤치마크 범위 · 각 정량결론 **확신도·N/A**) · **변경이력 append**(Business 방법론 부록) · Business update(plm-sync 자동 upsert) · `state.tasks.dig.status` 종단(Go = `"ready (→plan)"` / No-Go = `"closed (no-go)"` / Pivot = `"awaiting-feedback (pivot·재조사)"`) · current.md.

## 판정 게이트 (§4 인라인 transcribe — 이 스킬 본문이 런타임 권위)

### Go/No-Go/Pivot 결정트리 (배타·완전·우선순위 순 · 출시전 관측입력만)
> 핵심: **N/A(증거부재)와 반증(불리 관측)을 구분**. N/A는 kill이 아니다. 구조적 비적용(예: 광고모델의 결제수수료) = 실패 아님 · 0 처리 · 게이트 안 죽임. 판정은 **시장측(§1·§4·§5)·경쟁측(§2·§3) 양쪽 입력**의 종합.

1. **① 판정유보 (Pivot·추가조사)** — if 핵심 게이트입력(**필요점유율 · 수요프록시 · 단위기여마진**) 중 **근거불충분(evidence gap)** ∨ 핵심입력 **저확신**.
2. **② No-Go (closed)** — elif **실체 반증**: `필요점유율 > SOM현실` ∨ `수요프록시가 관측근거로 미충족` ∨ `전 후보 단위기여마진 ≤ 0(관측)`.
3. **③ Pivot (모델/포지션 전환)** — elif `방어가능 차별점 = 0`(시장·수요·마진 OK인데 차별화 없음).
4. **④ Go** — else: `방어가능 차별점 ≥ 1 ∧ 수요프록시 충족 ∧ 단위기여마진 > 0 후보 ≥ 1 ∧ 필요점유율 ≤ SOM현실 ∧ 핵심입력 근거충분`.

- **수요프록시(관측)** = `직접 경쟁 ≥ 1 유료전환 관측 ∧ 미충족 JTBD 2차근거 ≥ 2 독립출처`.
- **LTV/CAC = 게이트 아님** → 사후 KPI. **저확신 No-Go 금지**(①이 흡수) — 저확신 Go 금지와 대칭.

### 확신도 규칙
각 게이트입력에 확신도 `{높/중/낮/근거불충분}`. 핵심 게이트입력이 근거불충분/저확신 → 트리 ①(판정유보). **구조적 비적용(0)** 과 **근거불충분(evidence gap)** 은 구분(비적용 셀은 게이트를 죽이지 않음).

### 단위기여마진 공식 (수익화 맥락)
`단위기여마진 = 가격 − 플랫폼 수수료 − 결제 수수료 − 유저당 가변원가(추론/인프라/지원)`. 각 항에 **근거|가정** 표기. 눈금: **1 = ≤0**(수수료 + COGS가 가격 잠식) / **2 = 0~30%** / **3 = >30%**.

### 삼각검증 (WebSearch GIGO 방어)
각 주장 = **2 독립출처**(동일 원출처 재인용 = 1). 출처유형 명시(스토어 · 리뷰 · 리포트 · 커뮤니티). **WTP 현시선호(경쟁 실지불가) 1순위**, 표명선호(reddit "too expensive")는 방향성 · 확신도 상한 '중간'.

### 눈금 앵커 (요약 — 전체 서술자·근거셀은 business.json 동봉)
- **방어가능성 (CA 차별화 맥락)**: 상 = 데이터 독점 / 네트워크 효과 / 규제 라이선스 · 중 = 전환비용 / 브랜드 / 통합 깊이 · 하 = UI 카피 1스프린트.
- **적합성 혼합축(1/2/3) (수익화 맥락)**: 타깃WTP(추론) · 시장가격대 진입 · 전환마찰 · 단위기여마진 — 각 1/2/3 판별문은 business.json §4.2.
- **체험 루브릭(1/3/5) (체험 맥락)**: 온보딩마찰 · 핵심작업시간 · 리텐션훅 · 페이월위치 · 마찰점 · delight — 각 1/3/5 판별문은 business.json §2.4. **각 점수엔 근거셀**(출처/체험 인용).
