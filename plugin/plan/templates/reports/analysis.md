# 분석(analysis) 보고서 가이드 — 요구 정리·감사·설계 검토

> 용도: 접수된 요구 전수 정리, 문서↔구현 정합 감사, 설계 대안 비교 등 **판단 근거를 구조화**해 의견 수렴/결정을 준비. `_COMMON.md` 선행.

## 핵심 원칙
- **누락 0**: 입력(사용자 메시지·감사 대상)의 전 문장을 항목 카드에 매핑. 부록 검증 로그에 "원문 묶음 N건 → 카드 M건" 집계 명시.
- **원문 보존**: 각 카드에 사용자 원문을 인용 블록(보라 좌보더)으로 그대로 싣고, 그 아래 해석·현재 사실을 분리 서술 — 해석이 원문을 대체하지 않게.
- **사실은 증거와**: "현재 사실" 절은 코드 경로·마이그레이션 번호·검증 결과 등 확인 가능한 근거만. 추정은 "추정/조사 필요"로 명시.

## 섹션 구성
1. **Executive Summary** — 무엇이 들어왔고, 어디에 결정이 필요한지, 기존 자산과의 관계(개정 N·신규 M). KPI 카드(항목 수·매핑 수·갭 수·질문 수).
2. **항목 전수 카탈로그** — ID 부여(A1, B2…, 피드백 참조용), 카드마다: 원문 인용 → 해석 → 현재 사실 → 상태 칩(구현됨/Draft 기획/개정 필요/부분/신규 갭).
3. **기존 자산 매핑 표** — 신규 요구 ↔ 기존 아티팩트, 조치(개정/신규/무관) — 중복 발급 방지.
4. **쟁점·질문** — 결정이 필요한 지점을 선택지 표(장단점)+추천+근거로. (수가 많으면 survey 타입으로 분리 발행.)
5. **조사 예정** — 질문 없이 진행 가능한 항목과 그 계획.
6. **제안 로드맵** — 단계 패키지와 터치 포인트.
7. **리스크** — 번호 붙여 우선순위화.
8. **부록** — 출처·검증 로그.

## 상태 칩 어휘 (일관 사용)
`구현됨(녹)` `Draft 기획(황)` `기획 개정 필요(보라)` `부분 존재(청)` `신규 갭(적)` `OK(녹)`
