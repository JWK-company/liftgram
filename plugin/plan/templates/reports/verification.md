# 검증(verification) 보고서 가이드 — E2E·라이브 테스트 증명

> 용도: "고쳤다/된다"를 **증거로 증명**하는 보고서(배포 검증, 버그 수정 확인, E2E 실증). `_COMMON.md` 선행.

## 핵심 원칙
- **주장 금지, 증거만**: 모든 "성공"에는 재현 가능한 증거를 병기 — 실행 커맨드, HTTP 상태·응답 본문 발췌, DB 행, 스크린샷(업로드 후 참조), 로그 라인.
- **재현 커맨드 그대로**: 검증에 쓴 커맨드를 코드 블록으로(시크릿은 마스킹) — 제3자가 따라 할 수 있게.
- **실패·제약도 기록**: 검증 못 한 항목은 "미검증+사유"로 명시(조용한 생략 금지). 부분 성공은 부분으로.

## 섹션 구성
1. **Executive Summary** — 검증 대상·결과 한 줄(성공 N/전체 M)·잔여 리스크.
2. **검증 매트릭스** — 표: | # | 검증 항목 | 방법 | 기대 | 실측 | 판정(✅/⚠/❌) | — 판정은 실측 근거와 1:1.
3. **항목별 상세** — 각 항목: 절차(커맨드) → 실측 출력(발췌) → 판정 근거. 스크린샷은 캡션에 확인 포인트 명시.
4. **발견·수정 사항** — 검증 중 발견한 버그와 처리(수정 커밋·재검증 결과).
5. **미검증·제약** — 환경 제약으로 남은 것과 후속 계획.
6. **부록** — 환경(호스트·버전·시각), 정리(테스트 리소스 삭제 여부), 검증 로그.

## 주의
- 테스트 리소스(계정·프로젝트·임시 인증 설정)는 검증 후 **반드시 정리**하고 정리 사실을 부록에 기록.
- 시각은 서버 기준으로(호스트 간 시계 스큐 주의 — 실측 사례: 로컬↔서버 81초 차).
